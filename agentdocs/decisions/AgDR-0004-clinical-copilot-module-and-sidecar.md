# Build Clinical Co-Pilot as a custom OpenEMR module + Python sidecar

> In the context of shipping a verifier-gated clinical co-pilot inside OpenEMR for the AgentForge sprint,
> I decided to scaffold a `oe-module-clinical-copilot` custom module (PHP) plus a separate `agent/copilot-api` FastAPI sidecar (Python),
> accepting a two-process architecture and a shared-secret hop,
> to achieve a clean trust boundary where the sidecar never holds MariaDB credentials, only sees source packets the gateway hands it, and can use Python's stronger LLM/eval ecosystem.
> Alternatives considered: PHP-only orchestration (rejected: weaker eval/observability ecosystem); single combined service (rejected: blurs the credential boundary); generic AI panel via existing dashboard-context module (rejected: dashboard-context is for widget visibility, not LLM gating).

## Consequences

- Two services to deploy on Railway: `openemr` (web) and `copilot-api` (private). The sidecar must have **no public domain**.
- The gateway in PHP is the trust boundary — session, CSRF, ACL, server-side `pid`, audit logging all happen there before any sidecar call.
- The sidecar is reachable only via `X-Copilot-Gateway-Secret` shared secret (`COPILOT_OPENEMR_GATEWAY_SHARED_SECRET`).
- LLM is `claude-sonnet-4-6` with adaptive thinking and prompt caching on the system prompt, structured output via `messages.parse(output_format=LLMOutput)`. Repair-once on verifier failure.
- Verifier is intentionally strict (8 rules); strictness is the feature.
- `trace_id` (UUIDv4) is generated by the PHP gateway and threaded through the sidecar request, Langfuse spans, and the `audit_master` row, giving a single pivot key for incident response.

## Verification

- 7 source packets returned for demo patient `pid=1` (Phil Belford) via authenticated POST to `/interface/modules/custom_modules/oe-module-clinical-copilot/public/api/brief.php`.
- Eval suite `python -m evals.runner` passes 5/5 (groundedness, blank-vs-negative, cross-patient, refusal-scope, unsupported-source).
- Module loads through `ModulesApplication::bootstrapCustomModules()` (registered with `mod_active=1`, `type=0`).
