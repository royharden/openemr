## 2. Product Decision: Free Text, But Not a Blank Chatbot

The free-text UI should satisfy `planning/Users.md` without betraying it.

Dr. Sarah Clark has 90 seconds. She should not land on an empty chatbot and invent prompts. The first screen remains the automatic pre-room brief plus high-value buttons. Free text appears only after that as a **follow-up affordance** for questions the buttons do not cover:

- "Did she fill her Metformin refill?"
- "When was the last tetanus shot?"
- "Any abnormal labs since March?"
- "What dose of Lisinopril is she on?"
- "Anything new since I last saw her?"

The free-text box must be explicitly patient-scoped, read-only, and verifier-gated. It should feel like asking the chart a narrow question, not asking an AI doctor.

### UX Shape

Add a compact input row below the suggested action buttons:

- Placeholder: `Ask about this patient's chart...`
- Submit button: `Ask`
- Small helper text: `Current patient only. Source-cited answers.`
- Enter submits; Shift+Enter optionally creates a newline if using a textarea.
- Disable while running; show "Checking chart..." and preserve the prior answer until the new one arrives.
- Render the response in the same verified-claims format as button follow-ups.
- If the query is out of scope, render a refusal line and keep the chart usable.

Do not create a separate chat page, modal, or hero surface.

## 3. Interaction Contract

The free-text turn should be a new use case, not an unbounded backdoor.

### Request

From browser to PHP gateway:

```json
{
  "use_case": "free_text_followup",
  "question": "Did she fill her Metformin refill?",
  "conversation_id": "optional browser-generated UUID for this chart session"
}
```

Server facts continue to be server-owned:

- `pid` from OpenEMR session, not request body.
- `patient_uuid` from gateway lookup.
- `patient_uuid_hash` generated by gateway.
- `trace_id` generated by gateway.
- `user_id` from session.
- allowed tools/builders selected by application code, not model output.

### Response

Reuse `VerifiedResponse`:

```json
{
  "answer_type": "follow_up",
  "claims": [
    {
      "text": "Metformin 500 mg BID is active on the medication list.",
      "claim_type": "fact",
      "source_ids": ["rx:prescriptions:101"],
      "caveat": null
    }
  ],
  "missing_data": ["Dispense/fill records were not retrieved in this version."],
  "refusals": [],
  "suggested_followups": ["Medication check", "Recent abnormal labs"],
  "verifier_status": "passed_with_drops",
  "trace_id": "..."
}
```

If the question asks for diagnosis, treatment, prescribing, order entry, broad search, or another patient:

```json
{
  "answer_type": "refusal",
  "claims": [],
  "missing_data": [],
  "refusals": ["I can only answer read-only questions about the currently open patient's chart."],
  "suggested_followups": ["Medication check", "Recent abnormal labs"],
  "verifier_status": "passed",
  "trace_id": "..."
}
```

## 4. Routing Strategy

Do not let the LLM decide which OpenEMR data it may access. The PHP gateway should classify the free-text question into one of the existing safe packet-builder bundles.

### Initial Deterministic Router

Add a small PHP or Python router using keyword/domain matching:

| Query family | Example | Packet builders |
|---|---|---|
| Medication/refill/adherence | "Did she fill Metformin?" | identity, active medications, allergies |
| Allergy | "Any allergy issue?" | identity, allergies, active medications |
| Labs/results | "Any abnormal labs?" | identity, active problems, recent labs |
| Immunization/preventive | "Last tetanus?" | identity, immunizations |
| What changed | "Anything new since March?" | identity, problems, meds, allergies, recent labs, immunizations |
| Identity/basic fact | "How old is she?" | identity |
| Unknown but plausible chart question | fallback | full v1 bundle capped at 50 packets |
| Out-of-scope clinical action | "Should I increase lisinopril?" | no sidecar call; return local refusal |

This keeps the feature useful without inventing tool-chaining complexity under deadline pressure.

### Later Upgrade

After deployment works, replace keyword routing with a tiny structured intent classifier only if evals show keyword routing misses common doctor questions. The classifier must output an enum only; it still cannot request arbitrary patients or arbitrary data.

## 5. Conversation Memory

The PRD asks for multi-turn context, but clinical safety argues against a long hidden chat transcript.

Use **short, explicit chart-session context**:

- The current patient remains the hard boundary.
- Store only the last 3 user questions and response summaries in browser memory for display.
- Send to sidecar:
  - current question
  - current use case
  - latest verified claims from the previous turn, if needed
  - no raw previous model prose unless it was verifier accepted
- Do not persist the full conversation in a new database table for v1.
- Continue writing one `agent_turn` audit row per turn with trace ID.

This satisfies "maintain context across a conversation" enough for Week 1 while avoiding an unmanaged PHI chat log.

Example:

1. User: "Medication check."
2. Agent: verified meds/allergies.
3. User: "What about Metformin?"
4. Sidecar receives the current question plus prior accepted source IDs/claims and the medication/allergy packet bundle.

## 7. Other Local Gaps Worth Closing

These are not all blockers for a local demo, but they affect grading/interview defensibility.

### A. Source Chips Are Not True Chart Links Yet

Status file says chips render but deep-link behavior is unconfirmed. Current JS renders source IDs as chips; it does not appear to open the underlying OpenEMR record.

Plan:

- Add a source drill-down endpoint or in-panel packet details popover.
- For Week 1, a safe packet-details popover is enough if deep OpenEMR record URLs are fiddly.
- Avoid leaking raw SQL/table IDs into URLs beyond the current session.

### B. Sidecar Does Not Validate Task Token

PHP sends `X-Copilot-Task-Token`, but sidecar auth currently validates only `X-Copilot-Gateway-Secret`.

Plan:

- Add sidecar validation for HMAC token signature and expiry.
- Check scope is `read-only`.
- Check token patient hash matches `BriefRequest.patient_uuid_hash` indirectly or include patient hash in token payload.
- Add tests for expired, tampered, and wrong-scope tokens.

This is a security hardening item and a good interview talking point.

### C. Feedback Is Langfuse + Audit, Not a Dedicated Feedback Table

`planning/Users.md` says feedback writes to Langfuse and a feedback table. Current implementation writes Langfuse score and an `agent_turn` audit row.

Plan:

- Either update `Users.md` to describe the implemented v1 truth, or add a small local feedback persistence table.
- For Week 1, I would update the doc: audit + Langfuse is sufficient and less schema-invasive.
- For v2, create `clinical_copilot_feedback` with trace ID, verdict, user ID, pid, timestamp, and no raw PHI comment by default.

### D. Preventive Care Is Immunizations-Only

The UI and docs mention preventive gaps. Current packet builder covers immunizations; it does not implement age/sex guideline logic or mammogram/CRC screening.

Plan:

- Rename UI/README language to "Immunizations" if no guideline logic is implemented.
- Or add a modest "immunization history" follow-up, not "preventive gaps."
- Leave USPSTF/ACIP rules for v2 unless there is time to implement and eval them properly.

### E. Direct SQL in Packet Builders vs Earlier Architecture Wording

The implementation uses parameterized OpenEMR SQL APIs inside the gateway. Earlier planning preferred services/REST/FHIR and said direct SQL was forbidden in agent code.

Plan:

- Make the architecture wording precise: direct SQL is forbidden in the sidecar/LLM path; parameterized gateway SQL is accepted for Week 1 where OpenEMR service coverage is thin.
- Keep "sidecar has no DB credentials" as the real trust boundary.

## 9. Grading Posture

If free-text follow-up is added as above, the local project will be much easier to defend against the PRD:

- Agentic chatbot: automatic brief + buttons + free-text follow-up + short chart-session context.
- Verification: still every claim goes through deterministic verifier.
- Observability: every free-text turn gets trace/token/cost/feedback metadata.
- Evaluation: free-text has clinical edge-case evals, not just happy path.
- User fit: Dr. Clark does not have to prompt from a blank screen, but can ask natural follow-ups when the briefing misses her immediate question.

The strongest defense line:

> The chatbot is intentionally constrained. It is conversational where conversation helps Dr. Clark ask patient-specific follow-ups, but it remains read-only, current-patient-bound, source-cited, and verifier-gated because this is a clinical context.
